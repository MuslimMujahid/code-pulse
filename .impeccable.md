## Design Context

### Users
Software developers using CodePulse locally while actively working on a codebase. Technical professionals who read code for a living. Their job to be done: discover hidden patterns in git history — file coupling, churn hotspots, contributor ownership, and codebase evolution over time — that are invisible in a flat commit list. They are comfortable with density and data; they do not need hand-holding.

### Brand Personality
**Alive, Precise, Revealing.** The interface should feel like observing something real and complex from a distance — a neural network firing, a city's nervous system lighting up at night. Technical and analytical, but visually cinematic. Never playful, never corporate.

### Aesthetic Direction
Dark mode only. Background: deep space dark (`#0d0d1a`). Nodes: electric blue to violet gradient (`#3b82f6` → `#8b5cf6`). Active/recent edges: amber (`#f59e0b`). Dormant edges: muted slate. The canvas is the hero — a dense force graph that rewards exploration. Think: satellite imagery of a city at night, not a SaaS dashboard.

Anti-references: GitHub's flat commit list UI, bright enterprise dashboards (Jira, Linear light mode), generic white developer tools. No card grids, no pastel palettes, no excessive whitespace.

Reference feel: Observable notebooks, Grafana dark theme, Three.js demos — tools that feel like instruments, not apps.

### Design Principles
1. **Data speaks first** — every visual element must encode real data. Node size = commit frequency. Edge thickness = co-change count. Color = recency or contributor. No decoration without information.
2. **Motion reveals meaning** — animation communicates temporal causality (timeline replay) and state transitions. Animation is never decorative; it always answers "what changed and why."
3. **Dark, dense, readable** — high contrast text and glyphs on dark backgrounds. Embrace density like a terminal or a data-viz tool. Avoid excessive padding and empty space.
4. **Developer-grade precision** — tooltips show exact numbers. Labels use technical naming (file paths, commit SHAs, author emails). No marketing copy, no onboarding fluff. The audience reads stack traces for fun.
5. **One canvas, multiple lenses** — the force graph is the single source of truth. Heatmap, contributor, and coupling views are color/size overlays on the same graph, never separate pages or tabs.
